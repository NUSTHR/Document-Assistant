更新时间: 2026-04-23 真实联调执行计划

十一、接入真实 RAGFlow 的具体做法
1. 先拿到真实可用的 RAGFlow 服务访问地址，填入 `ragflow_integration_service/.env` 的 `RAGFLOW_BASE_URL`。
2. 再拿到真实可用的 RAGFlow API Key，填入 `ragflow_integration_service/.env` 的 `RAGFLOW_API_KEY`。
3. 这里的 `RAGFLOW_BASE_URL` 必须是 RAGFlow 对外可访问的服务根地址，当前代码会自动拼成 `.../api/v1`，因此 `.env` 中不要再手动追加 `/api/v1`。
4. 修改 `.env` 后，必须重启 `ragflow_integration_service`，否则进程仍会继续使用旧配置。

十二、后端重启后的第一轮验证
1. 先验证健康检查 `GET /health`，确认 integration-service 自身仍然正常启动。
2. 健康检查通过后，再验证 integration-service 是否能真正连通底层 RAGFlow。
3. 这一轮的目的不是看前端页面，而是先确认“配置正确、服务可连、鉴权可过”。
4. 如果这一步失败，优先排查：
   - `RAGFLOW_BASE_URL` 是否填成了错误地址
   - `RAGFLOW_API_KEY` 是否无效或权限不足
   - integration-service 所在机器是否能访问 RAGFlow 服务

十三、`POST /api/files` 的真实验收方式
1. 准备一个真实测试文件，内容尽量短且可预期，例如一份只有几段固定文本的 `txt` 或小型 `pdf`。
2. 选定一个测试知识库名称，例如 `finance-kb-test`。
3. 选定一个业务文件 ID，例如 `FILE-TEST-0001`。
4. 选定一个业务文件名，例如 `test-report.txt`。
5. 调用 `POST /api/files` 上传文件并触发解析。
6. 验证返回结果中：
   - `biz_file_id` 是否仍然是 `FILE-TEST-0001`
   - `biz_file_name` 是否仍然是 `test-report.txt`
   - `parse_status` 是否进入完成态
   - `chunk_count` 和 `token_count` 是否大于等于合理值
7. 这一步的核心不是只看接口是否 `200`，而是确认防腐层埋点逻辑确实成立：文件进入 RAGFlow 后还能被正确反解回业务字段。
8. 如果上传成功但解析失败，需要重点检查：
   - RAGFlow 是否支持该文件格式
   - 底层解析模型或解析能力是否可用
   - 当前 SDK 返回的 `Document.run` / `status` 是否与代码中的完成态判断一致

十四、`POST /api/chat` 的真实验收方式
1. 在 `POST /api/files` 验证通过后，再使用同一个知识库语料进行提问。
2. 为 `assistant_name` 选定一个固定值，例如 `finance-assistant-test`。
3. 问题内容必须直接命中刚刚上传文件中的文本，避免因为检索不到而误判代码错误。
4. 调用 `POST /api/chat`，观察返回的 `text/event-stream` 数据块。
5. 验证流式回答中：
   - `answer` 是否持续返回文本片段
   - 回答中的内部引用标记是否已被替换为 `[^1]` 这类脚注样式
   - `references` 是否返回数组
   - 每个引用对象中是否都带有 `biz_file_id`、`biz_file_name`、`chunk_content`、`similarity_score`
6. 这一轮的核心是确认当前防腐层对 RAGFlow SSE 报文结构的假设是否正确，特别是：
   - `session.ask(..., stream=True)` 返回的 `message.content`
   - `message.reference`
   - `reference.document_name`
7. 如果回答能出来但 `references` 为空，优先怀疑：
   - 当前 RAGFlow 实例没有返回引用块
   - SDK 在当前版本下对流式 `reference` 的封装字段与本地读取方式不一致
8. 如果 `answer` 里仍然残留 `##0$$` 或 `[ID:0]`，说明当前正则清洗规则还不完整，需要根据真实报文补充模式。

十五、联调通过的判断标准
1. `POST /api/files` 返回业务字段，且解析结果正常。
2. `POST /api/chat` 能稳定返回流式回答。
3. 流式回答中的引用标记已被替换为标准脚注格式。
4. `references` 中不再出现底层 `dataset_id`、`document_id`，而是完整业务字段。
5. 即使个别引用反解失败，也只能降级为 `UNKNOWN`，不能让整段回答直接中断。

十六、当前最合理的执行顺序
1. 先填真实 `.env`
2. 再重启 integration-service
3. 先验 `GET /health`
4. 再验 `POST /api/files`
5. 最后验 `POST /api/chat`
6. 只有这三步都过，才能说明当前防腐层假设已经被真实 RAGFlow 实例验证

四十、Verba 在本次评估中的真实定位
1. Verba 不是一个单纯可替换的“RAG 前端皮肤”。
2. Verba 官方说明它本身是一套完整 RAG 应用，包含：
   - `Next.js` 前端
   - 自带 `FastAPI` 后端
   - 围绕 `Weaviate` 构建的检索/入库/生成链路
3. Verba 官方文档还明确说明：
   - 前端请求是发给它自己，再由它自带的 FastAPI 处理
   - 当前 FastAPI 主要用于其内部通信，不推荐当作外部公共 API 使用
4. 同时，Verba 官方 FAQ 明确说明它不能直接复用既有 Weaviate 数据，因为它要求自己的特定数据格式。
5. 这意味着 Verba 的设计前提是“Verba 主导整套 RAG 栈”，而不是“拿它的前端去挂接别人的 RAG 后端”。

四十一、把 Verba 集成进当前项目的难度评估
1. 难度不是“小改可接”，而是中高难度。
2. 难点不在页面样式，而在系统边界完全不同：
   - 当前项目链路是 `Vue 3 + Vite -> ragflow_integration_service -> RAGFlow`
   - Verba 链路是 `Next.js 前端 -> Verba FastAPI -> Weaviate/LLM/其自有 RAG 组件`
3. 如果强行接入 Verba，就不是“接一个前端”，而是要处理两套后端模型、两套数据摄入逻辑、两套检索契约和两套运行时依赖。
4. 当前项目的核心设计原则是“RAGFlow 作为底座，integration-service 作为唯一防腐层”，而 Verba 默认设计会绕开这条边界。
5. 因此，如果目标是保持当前项目架构克制、低耦合、可维护，那么直接把 Verba 整套塞进来并不自然。

四十二、如果一定要集成 Verba，唯一合理的技术路径
1. 不能把 Verba 当完整 RAG 应用直接并入主链路。
2. 唯一合理路径是：
   - 只借用 Verba 的前端交互设计或前端局部组件思路
   - 不复用 Verba 的 Weaviate 数据层
   - 不复用 Verba 的 FastAPI 作为业务后端
   - 继续坚持当前 `ragflow_integration_service` 作为唯一后端入口
3. 更具体地说，技术路径应该是“参考式集成”而不是“原样集成”：
   - 研究 Verba 的文档展示、引用区块、对话区、检索片段呈现方式
   - 在当前 `D_ass` 中重建对应的前端交互
   - 所有数据仍来自 `/api/files` 和 `/api/chat`
   - 如果需要扩展文档预览能力，应在当前防腐层上新增明确 DTO，而不是去对接 Verba 自带 FastAPI
4. 如果硬要直接复用代码，理论上也只能走“抽取 Verba 前端页面层并重写其 API 适配层”这一路，但这会涉及：
   - `Next.js` -> 当前 `Vue 3` 的技术栈迁移或并存
   - Verba 内部 API 调用逻辑的大量重写
   - Weaviate 假设到 RAGFlow 假设的整体替换
5. 这条路工作量明显大于在当前前端中直接实现“文档引用展示 + 文档预览 + 溯源核对”。

四十三、为什么当前不推荐直接集成 Verba
1. Verba 的核心依赖是 Weaviate 生态，而当前项目底座已经确定为 RAGFlow，底层范式不同。
2. Verba 不是纯前端组件库，而是自带完整后端和数据流，接入后会冲击当前已建立的防腐层边界。
3. 当前项目前端技术栈是 `Vue 3 + Vite`，Verba 前端是 `Next.js`，直接复用的技术成本高。
4. Verba 官方自己也说明当前项目偏社区性质维护，不能按生产级稳定组件对待。
5. 你当前真正想解决的问题是“让用户更方便地查看引用文档、核对答案”，这并不要求引入整套 Verba 技术栈。
6. 这个目标在当前前端内直接增强引用展示、文档预览、答案-片段对照能力，成本更低、边界更干净、与现有架构更一致。

四十四、最终结论
1. 当前 RAGFlow 功能还不能直接宣称“已经完全打通可用”，因为本项目真实 `.env` 中的 `RAGFLOW_BASE_URL` 仍未改成正确入口。

   `RAGFLOW_BASE_URL=http://127.0.0.1:18080`
   正式回填进 `ragflow_integration_service/.env`。
3. 对 Verba 的结论是：不推荐直接整体集成。
4. 如果只是欣赏 Verba 的交互效果，推荐做法是“参考其前端交互设计，在当前 `D_ass` 中重建文档展示与答案核对界面”，而不是把 Verba 的整套 Weaviate-RAG 应用直接接入当前项目。

八十五、前端 `D_ass` 的当前定位

   - 中间：助手名称、会话名称、问题输入、流式回答展示、回答取消
   - 右侧：引用卡片、激活引用、文件 chunk 预览、原文加载/预览/下载
3. 前端配置入口在 `src/config/app-config.ts`：
   - 默认后端地址：`http://localhost:8081`
   - 普通请求超时：`10000ms`
   - SSE 连接超时：`15000ms`
4. `src/lib/integration-api.ts` 对后端响应做了较严格的运行时结构校验，避免后端返回格式漂移时静默失败。
5. `src/lib/api-client.ts` 已经区分了 timeout、network、http、invalid_response、stream 等错误类型，并映射成前端可展示的信息。

八十六、后端 `ragflow_integration_service` 的当前定位
1. 后端是一个明确的 RAGFlow 防腐层，而不是直接把 RAGFlow 物理 ID 暴露给前端的薄代理。
2. FastAPI 入口在 `app/main.py`，当前注册：
   - `GET /health`
   - `/api/files` 系列文件接口
   - `POST /api/chat` 流式聊天接口
3. 配置在 `app/core/config.py`，通过 `.env` 读取：
   - `RAGFLOW_BASE_URL`
   - `RAGFLOW_API_KEY`
   - `RAGFLOW_TIMEOUT_SECONDS`
   - CORS 与应用端口等配置
4. 文件链路主要在 `RagflowKnowledgeBaseAdapter`：
   - 按 `knowledge_base_name` 查找或创建 dataset
   - 上传时把业务文件 ID 写入 RAGFlow document display name，格式为 `[biz_id:xxx]文件名`
   - 对前端返回 `biz_file_id`、`biz_file_name`、解析状态、chunk 数和 token 数
   - 支持文件列表、文件详情 chunk 预览、原文下载
5. 聊天链路主要在 `RagflowConversationAdapter`：
   - 不再完全依赖旧 SDK 的 `list_chats()` 假设
   - 当前手动调用 `/chats` 兼容 `data={ chats, total }` 与旧数组结构
   - 当前手动调用 `/chat/completions`，请求体使用 `messages`
   - 解析 RAGFlow SSE 数据块，累计 answer 后再输出给前端
   - 将 RAGFlow 内部引用标记清洗为 `[^1]` 形式
   - 从引用的 `document_name/docnm_kwd` 反解业务文件 ID 与业务文件名
6. `/api/chat` 的生成器在 SSE 已建立后遇到 RAGFlow 错误时，会输出 `ERROR: ...` 结构化消息并发送 done 事件，避免直接炸断前端流读取。
